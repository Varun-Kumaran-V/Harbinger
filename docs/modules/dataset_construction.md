# Dataset Construction & Feature Engineering Module

## Module Overview

The **Dataset Construction & Feature Engineering Module** transforms independent synthetic infrastructure signals into a machine learning dataset that can be used to train, validate, and benchmark Harbinger.

Unlike the previous subsystem, which generates **individual signals**, this subsystem combines those signals into a unified timeline, verifies that meaningful patterns exist, and engineers temporal features that improve the predictive capability of machine learning models.

The output of this module is **not a prediction**. Instead, it is a carefully prepared dataset that serves as the foundation for all downstream modelling.

---

# Why This Module Exists

Machine learning models cannot operate directly on disconnected telemetry streams.

Instead, they require:

- synchronized observations,
- meaningful labels,
- engineered temporal features,
- validated learnability.

This subsystem performs those responsibilities.

Without this module, Harbinger would have no structured dataset suitable for prediction.

# File 1 — build_training_dataset.py

## Purpose

Constructs the complete synthetic training dataset by combining every generated infrastructure signal into a single synchronized timeline.

This is the **dataset assembly engine** of Harbinger.

---

## Why It Exists

Each generator only models one aspect of cluster behaviour.

For example,

Checkpoint Generator

↓

Checkpoint signal

ECC Generator

↓

ECC signal

Temperature Generator

↓

Temperature signal

Machine learning requires all of these observations to exist together for the same point in time.

This file performs that integration.

---

## Inputs

Consumes outputs generated by:

- checkpoint_generator.py
- ecc_generator.py
- network_generator.py
- temperature_generator.py

along with internally simulated infrastructure metrics such as:

- GPU utilization
- CPU utilization
- Memory pressure
- Failure timeline
- Warning windows

---

## Processing

### Step 1

Generate a simulation timeline.

Example

```
Minute 1

Minute 2

...

Minute 10000
```

---

### Step 2

Generate or retrieve every infrastructure signal.

Examples:

Temperature

ECC

Checkpoint

Network

GPU utilization

CPU utilization

Memory pressure

---

### Step 3

Synchronize every signal.

Each row now represents the complete cluster state for one minute.

---

### Step 4

Assign labels.

Typical labels include:

```
Healthy

↓

Failure approaching

↓

Failure
```

These labels become the prediction targets.

---

## Outputs

### training_dataset.csv

The raw machine learning dataset.

Each row represents one point in simulated time.

Each column represents one telemetry feature.

---

## Why This Output Matters

Everything downstream depends on it.

If this dataset is poor,

every model will also perform poorly.

---

## Consumer

```
learnability_audit.py

and

engineer_and_audit.py
```

---

## Presentation Notes

If asked

"Why not train directly from generators?"

Answer

Because machine learning requires synchronized observations.

Generators produce independent signals.

This file combines them into one structured dataset.

# File 2 — learnability_audit.py

## Purpose

Determines whether the synthetic dataset actually contains learnable patterns.

Think of this as a **dataset quality check** rather than a prediction model.

---

## Why It Exists

Creating synthetic data does not automatically produce a useful dataset.

The question becomes:

"Can machine learning distinguish healthy behaviour from impending failure?"

This file answers that question.

---

## Inputs

```
training_dataset.csv
```

---

## Processing

### Step 1

Load the raw dataset.

---

### Step 2

Perform a temporal train/test split.

Older observations become training data.

Later observations become testing data.

This prevents future information from leaking into the past.

---

### Step 3

Train several simple baseline models.

Examples include:

- Logistic Regression
- Decision Trees
- Random Forest (depending on implementation)

---

### Step 4

Measure predictive performance.

Metrics indicate whether useful patterns exist.

---

## Outputs

Console diagnostics and evaluation metrics.

No permanent repository artifact is produced.

---

## Why This Output Matters

If every baseline performs poorly,

either:

- the dataset is unrealistic,

or

- the labels contain little information.

This validation should occur before investing time in feature engineering.

---

## Consumer

Human researcher.

The results guide the next development step rather than feeding another script.

---

## Important Terms

Learnability

The extent to which useful predictive patterns exist.

Temporal Split

Training on earlier observations and testing on later observations.

Feature Leakage

Accidentally using future information during training.

---

## Presentation Notes

This module validates the **dataset**, not Harbinger itself.

It asks whether prediction is possible before attempting sophisticated modelling.

# File 3 — engineer_and_audit.py

## Purpose

Transforms raw telemetry into informative machine learning features.

---

## Why It Exists

Raw telemetry is often noisy.

Example:

```
GPU Utilization

87%
```

One instantaneous value provides little context.

Instead,

Harbinger generates features such as

Rolling average

Rolling variance

ECC count over 180 minutes

Temperature trend

These describe system behaviour over time.

---

## Inputs

```
training_dataset.csv
```

---

## Processing

### Step 1

Load the raw dataset.

---

### Step 2

Generate temporal features.

Examples include:

Moving averages

Rolling sums

Rolling standard deviations

Historical counts

Sliding windows

---

### Step 3

Remove redundant or invalid features if required.

---

### Step 4

Perform engineering diagnostics.

Ensures generated features are suitable for downstream modelling.

---

## Outputs

### engineered_training_dataset.csv

Contains machine learning features instead of raw telemetry.

---

## Why This Output Matters

Machine learning models generally learn much better from engineered temporal features than from instantaneous measurements.

This file converts telemetry into predictive information.

---

## Consumer

```
baseline_audit.py

tune_baselines.py

harbinger_pipeline.py
```

---

## Presentation Notes

If asked

"Why engineer features?"

Answer

Failures usually develop gradually.

Temporal statistics capture those gradual changes much better than single measurements.

# Common Questions

### Why generate temporal features?

Because failures evolve over time.

Historical behaviour is often more predictive than instantaneous measurements.

---

### Why perform learnability analysis first?

To verify the dataset itself is suitable before building sophisticated models.

---

### Why maintain both datasets?

Raw dataset

Represents simulated infrastructure.

Engineered dataset

Represents machine learning features.

Keeping both improves reproducibility and allows independent verification of feature engineering.
