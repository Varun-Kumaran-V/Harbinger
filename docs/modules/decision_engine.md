# Decision Engine Module

## Module Overview

The **Decision Engine Module** is the core contribution of Harbinger.

Everything developed before this point exists to answer one question:

> "Can we predict failures?"

This subsystem answers a much more important question:

> "Given a prediction, what should the cluster actually do?"

Traditional failure prediction systems stop after producing a probability.

Example:

```
GPU Failure Probability

82%
```

Harbinger continues one step further.

Instead of only predicting,

it determines whether intervention is economically worthwhile.

This transforms Harbinger from a prediction system into a **decision-support system**.

---

# Why This Module Exists

Predictions alone do not reduce downtime.

Suppose a model predicts

```
Failure Probability = 90%
```

Should we

- checkpoint?
- migrate the job?
- ignore it?
- continue training?

A prediction model cannot answer these questions.

Harbinger introduces cost-aware decision making.

Instead of maximizing prediction accuracy,

it minimizes expected operational cost.

This is the primary research contribution of the project.

Acts as the orchestrator of the Harbinger decision framework.

It coordinates prediction, cost estimation, decision making, and recommendation generation.

Think of it as the controller of the entire runtime pipeline.

## Why It Exists

Without this file,

every component would operate independently.

This pipeline connects:

Prediction

↓

Risk Assessment

↓

Cost Estimation

↓

Decision

↓

Recommended Action

Engineered telemetry features


Expected Failure Cost

vs

# File 2 — recovery_engine.py

## Purpose

Calculates how much training progress would be lost if a failure occurs.

---

## Why It Exists

Prediction probability alone cannot determine severity.

Example

Two failures:

Failure A

Loses

5 minutes.

Failure B

Loses

5 hours.

Both may have identical probability,

but vastly different operational impact.

Recovery Engine measures that impact.

---

## Inputs

Typically receives

Failure time

Checkpoint history

GPU count

Training progress

---

## Processing

### Step 1

Locate the most recent checkpoint.

---

### Step 2

Determine failure time.

---

### Step 3

Compute

Recovery Loss

```
Failure Time

−

Last Checkpoint
```

---

### Step 4

Convert lost time into

GPU-hours lost.

Example

```
40 minutes

×

64 GPUs

=

42.67 GPU-hours
```

---

## Outputs

Recovery loss.

GPU-hours lost.

Estimated work requiring recomputation.

---

## Why These Outputs Matter

Harbinger does not optimize prediction accuracy.

It optimizes

Recovered computation.

This module quantifies that objective.

---

## Consumer

```
intervention_cost_model.py
```

---

## Important Technical Terms

Recovery Loss

Training time that must be repeated.

GPU-Hours Lost

Total computational work wasted.

Checkpoint Recovery

Resuming training from the latest checkpoint.

---

## Presentation Notes

If asked

"Why calculate GPU-hours?"

Answer

Because losing one hour on a single GPU is very different from losing one hour across 128 GPUs.

GPU-hours provide a realistic operational cost.

# Common Questions

### Why isn't the highest probability always acted upon?

Because intervention itself has a cost.

Example

Failure probability

10%

Checkpoint cost

2 GPU-hours

Expected failure loss

0.5 GPU-hours

Checkpointing would actually increase overall cost.

---

### Why use Expected Cost?

Because uncertainty exists.

Harbinger balances

Probability

×

Impact

rather than probability alone.

---

### Why separate Recovery Engine and Cost Model?

Recovery Engine

Measures

"What is lost?"

Cost Model

Decides

"Is preventing that loss worthwhile?"

Keeping them separate improves modularity and allows different recovery or cost models to be substituted independently.

---

### Why have a Prediction Adapter?

It decouples prediction from decision making.

Future prediction models can replace the adapter without changing the downstream decision logic.

---

# Overall Mathematical Objective

Traditional systems optimize

```
Prediction Accuracy
```

Harbinger optimizes

```
Total Operational Cost
```

where

```
Total Operational Cost

=

Expected Failure Cost

+

Intervention Cost
```

The chosen action is the one that minimizes this value.

This shift in objective is the central innovation of Harbinger.

# Overall Module Summary

This subsystem transforms Harbinger from a machine learning model into an operational decision-support framework.

Its workflow is:

```
Failure Probability
        │
        ▼
harbinger_pipeline.py
        │
        ▼
Recovery Engine
        │
        ▼
Intervention Cost Model
        │
        ▼
Recommended Action
```

By the end of this subsystem, Harbinger has moved beyond answering:

> "Will a failure occur?"

and instead answers the more useful operational question:

> "Given the probability of failure, the expected recovery loss, and the cost of intervention, what action minimizes the total operational cost?"

This distinction—**prediction versus decision optimization**—is the key idea that differentiates Harbinger from conventional predictive maintenance systems and should be emphasized whenever presenting or defending the project.
